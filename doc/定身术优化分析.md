# 定身术（stasis_hex）双轨合并分析

> 目的：将 `being_frozen-arrow` / `being_frozen-area` 两种定身类型合并为统一的 `being_frozen` 系统  
> 日期：2026年5月7日  
> ⚠ 本文仅为分析，不做任何代码修改

### 1.1 两种触发方式

| | 箭击模式（Arrow） | 范围模式（Area） |
|---|---|---|
| **触发条件** | 玩家**非潜行**使用技能 | 玩家**潜行**使用技能 |
| **机制** | 发射一枚"定"弹射物，命中实体后定身 | 以玩家为中心，区域内所有生物被定身 |
| **定义文件** | `stasia_hex.json` → `projectile_data/ding.json` | `stasia_hex.json` → `area_select/main` |
| **等级** | 1~3 级（影响时长和叠加量） | 1~3 级（影响半径） |

### 1.2 当前双轨资源

| 资源类型 | Arrow 专用 | Area 专用 | 共享 |
|----------|-----------|----------|------|
| **计分板** | `wk.stasis_hex.freezeTimer-arrow` | `wk.stasis_hex.freezeTimer-area` | `freezeTimer-max_count`, `temp` |
| **实体标签** | `being_frozen-arrow` | `being_frozen-area` | `item_display.being_frozen`, `stasis_hex-area`(临时), `stasis_hex-ding`(弹射物) |
| **tick 处理** | `main-arrow.mcfunction` | `main-area.mcfunction` | `tick.mcfunction`(路由) |
| **效果应用** | `apply/arrow.mcfunction` | `apply/area.mcfunction` | `apply/apply_effects.mcfunction`, `display/main` |
| **效果移除** | `desctuor/remove_effects-arrow` | `desctuor/remove_effects-area` | `desctuor/auto_kill.mcfunction` |

## 二、两套系统的差异点逐行对比

### 2.1 初始时长设置

| | Arrow (`initial_hit`) | Area (`area_select/timer`) |
|---|---|---|
| **首次命中** | 等级1: 200t, 等级2: 300t, 等级3: 400t | 固定 300t |
| **叠加** | 等级1: +100t, 等级2: +150t, 等级3: +200t | 固定 +150t |
| **上限** | 1200t (60s) | 1200t (60s) |
| **最大叠加次数** | 10 次 (`freezeTimer-max_count`) | 10 次 (同一计分板) |

> 🔑 **核心差异**：箭击模式的时间与**弹射物等级**挂钩（可成长），范围模式的时间是**固定值**。

### 2.2 tick 计时器递减

| | `main-arrow.mcfunction` | `main-area.mcfunction` |
|---|---|---|
| **执行者** | `@e[tag=being_frozen-arrow]` | `@e[tag=being_frozen-area]` |
| **逻辑** | 减 `freezeTimer-arrow`，归零时调 `remove_effects-arrow` | 减 `freezeTimer-area`，归零时调 `remove_effects-area` |

> 🔑 两段代码**完全同构**，仅计分板名和 tag 名不同。

### 2.3 效果应用

| | `apply/arrow.mcfunction` | `apply/area.mcfunction` |
|---|---|---|
| **逻辑** | `freezeTimer-arrow / 20 + 1` → duration → `apply_effects` | `freezeTimer-area / 20 + 1` → duration → `apply_effects` |

> 🔑 两段代码**完全相同**，仅引用的计分板名不同。最终都调用同一个 `apply_effects`。

`apply/main.mcfunction` 的唯一作用是**按 tag 路由**到 arrow 或 area 函数：
```mcfunction
execute if entity @s[tag=being_frozen-arrow] run return run function ...apply/arrow
execute if entity @s[tag=being_frozen-area] run function ...apply/area
```

### 2.4 效果移除

| | `remove_effects-arrow` | `remove_effects-area` |
|---|---|---|
| **逻辑** | 恢复 AI、清除效果、移除 tag、重置计分板、杀展示实体 | 完全相同的操作 |

> 🔑 两段代码**完全相同**，仅 tag 名和计分板名不同。

### 2.5 叠加交互（Transfer 机制）

这是当前设计中最复杂的部分——当实体从一种定身类型切换到另一种时：

**箭击中已被范围定身的实体**（`arrow_hit/main.mcfunction`）：
```
1. 移除 being_frozen-area 标签
2. freezeTimer-arrow += freezeTimer-area  （转移剩余时长）
3. 重置 freezeTimer-area
4. 增加 freezeTimer-max_count
5. 检查 max_count ≤ 10
6. 叠加或初始击中的正常流程
```

**范围定身选中已被箭定身的实体**（`area_select/setting.mcfunction`）：
```
1. 移除 being_frozen-arrow 标签
2. freezeTimer-area += freezeTimer-arrow  （转移剩余时长）
3. 重置 freezeTimer-arrow
4. 增加 freezeTimer-max_count
5. 检查 max_count ≤ 10
6. 叠加或新定身的正常流程
```

> 🔑 这个 "转移剩余时长" 的逻辑是**双轨系统的最大复杂性来源**。如果合并为单一计时器，这些转移操作将完全不需要——计时器就是同一个，直接在上面叠加即可。

## 三、合并方案

### 3.1 统一后的资源

| 资源类型 | 合并前 | 合并后 |
|----------|--------|--------|
| **计分板** | `freezeTimer-arrow` + `freezeTimer-area` + `max_count` + `temp` = 4 个 | `freezeTimer` + `max_count` + `temp` = **3 个** |
| **实体标签** | `being_frozen-arrow` + `being_frozen-area` + 辅助标签 | `being_frozen` + 辅助标签 = **1 个主标签** |
| **tick 函数** | 2 个 main 文件 + tick 路由 | **1 个** main-tick 文件 |
| **apply 函数** | 3 个（main 路由 + arrow + area） | **1 个** |
| **desctuor 函数** | 2 个（arrow + area） | **1 个** |

### 3.2 需修改的文件

| 文件 | 改动类型 | 说明 |
|------|---------|------|
| `load.mcfunction` | 修改 | 将 `freezeTimer-arrow` + `freezeTimer-area` 合并为 `freezeTimer` |
| `tick.mcfunction` | 简化 | 2 条 execute 合并为 1 条；守卫也从 2 条简化为 1 条 |
| `main-arrow.mcfunction` | **删除** | 合并到新 `main-tick.mcfunction` |
| `main-area.mcfunction` | **删除** | 合并到新 `main-tick.mcfunction` |
| **新增** `main-tick.mcfunction` | 新建 | 统一的 tick 递减逻辑 |
| `arrow_hit/main.mcfunction` | 简化 | 移除转移逻辑（约 5 行） |
| `arrow_hit/initial_hit.mcfunction` | 微调 | 标签改为 `being_frozen` |
| `arrow_hit/superposition.mcfunction` | 微调 | 标签和计时器名改为统一版本 |
| `area_select/setting.mcfunction` | 简化 | 移除转移逻辑（约 5 行） |
| `area_select/timer.mcfunction` | 微调 | 计时器名改为 `freezeTimer` |
| `apply/main.mcfunction` | **删除** | 路由不再需要 |
| `apply/arrow.mcfunction` | **删除** | 合并到新 `apply/set_duration.mcfunction` |
| `apply/area.mcfunction` | **删除** | 合并到新 `apply/set_duration.mcfunction` |
| **新增** `apply/set_duration.mcfunction` | 新建 | 统一的时长计算 |
| `desctuor/remove_effects-arrow.mcfunction` | **删除** | 合并到新 `remove_effects.mcfunction` |
| `desctuor/remove_effects-area.mcfunction` | **删除** | 合并到新 `remove_effects.mcfunction` |
| **新增** `desctuor/remove_effects.mcfunction` | 新建 | 统一的清理逻辑 |
| `player_death/main.mcfunction` | 简化 | 2 次函数调用 → 1 次 |
| `display/main.mcfunction` | 不变 | 使用共享的 `item_display.being_frozen` 标签 |
| `arrow_hit/on_generic_arrow_entity.mcfunction` | 不变 | 弹射物标签 `stasia_hex-ding` 无需改动 |
| `desctuor/auto_kill.mcfunction` | 不变 | 使用弹射物标签，无需改动 |

### 3.3 文件数量变化

| | 合并前 | 合并后 | 净变化 |
|---|---|---|---|
| 总文件数 | 17 | 13 | **-4** |
| 删除 | — | 7 | |
| 新建 | — | 3 | |
| 修改 | — | 8 | |

### 3.4 核心逻辑简化

**最大的简化发生在叠加交互**——当前最复杂的 "转移" 逻辑被完全消除：

```
合并前 (arrow_hit/main.mcfunction 部分逻辑):
  tag @s remove being_frozen-area
  scoreboard players operation @s freezeTimer-arrow += @s freezeTimer-area
  scoreboard players reset @s freezeTimer-area
  scoreboard players add @s freezeTimer-max_count 1
  execute unless score @s freezeTimer-max_count matches ..10 run return run tag @s add being_frozen-arrow
  execute if score @s freezeTimer-arrow matches 1.. run return run function ...superposition
  function ...initial_hit

合并后:
  scoreboard players add @s freezeTimer-max_count 1
  execute unless score @s freezeTimer-max_count matches ..10 run return run tag @s add being_frozen
  execute if score @s freezeTimer matches 1.. run return run function ...superposition
  function ...initial_hit
```

**从 7 行核心逻辑简化为 3 行**，消除了：
- 转移剩余时长的 `scoreboard players operation` 操作
- 重置对方计时器的操作
- 切换标签的操作

## 四、风险与边界情况

### 4.1 ⚠️ 存量世界兼容性

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| 更新前已冻结的实体使用旧标签/计分板 | 更新后无法被新 tick 识别，效果永久残留 | 在 `load.mcfunction` 中添加一次性迁移函数：将所有 `being_frozen-arrow`/`being_frozen-area` 标签替换为 `being_frozen`，将旧计分板值迁移到新计分板 |

迁移伪代码：
```mcfunction
# load.mcfunction 末尾添加
execute as @e[tag=being_frozen-arrow] run tag @s add being_frozen
execute as @e[tag=being_frozen-area] run tag @s add being_frozen
tag @e remove being_frozen-arrow
tag @e remove being_frozen-area
# 迁移计分板分数（需要逐玩家/实体处理）
```

### 4.2 叠加次数上限

合并前后 `freezeTimer-max_count` 上限均为 10 次，行为不变。✅ 无风险。

### 4.3 最大时长上限

合并前后均为 1200t (60s)，行为不变。✅ 无风险。

### 4.4 展示实体（"定"字）

使用独立的 `item_display.being_frozen` 标签，与定身类型标签无关。✅ 无需修改。

### 4.5 弹射物自动清理

`auto_kill.mcfunction` 使用 `stasia_hex-ding` 弹射物标签，与定身类型标签无关。✅ 无需修改。

### 4.6 玩家死亡清理

合并后只需调用一次 `remove_effects`，减少了函数调用次数。✅ 更简洁。

### 4.7 两种触发方式的叠加量不同

这是一个有趣的边界情况。合并后叠加逻辑变为：
- 箭击叠加 → 调用 `arrow_hit/superposition` → 按等级 +100/+150/+200
- 范围叠加 → 调用 `area_select/timer` → 固定 +150

由于叠加量取决于**调用来源**（哪个 trigger 函数），而非被定身实体的标签，统一后不需要区分标签即可正确叠加。✅ 无风险。

### 4.8 初始时长来源不同

- 箭击初始时长 = 基于弹射物等级的查表（200/300/400）
- 范围初始时长 = 固定 300

统一后两个 trigger 函数各自设置初始值到同一个 `freezeTimer` 计分板。✅ 无冲突。

## 五、预期收益

### 5.1 性能

| 指标 | 当前 | 合并后 | 改善 |
|------|:---:|:-----:|:----:|
| tick.mcfunction 中的 stasis 相关指令 | 3 条（2 execute + 1 item_display） | **2 条**（1 execute + 1 item_display） | -33% |
| tick.mcfunction 守卫 | 2 条（防重复） | **1 条** | -50% |
| apply 路由检查 | 2 条 tag 检查 | **0 条** | -100% |

### 5.2 可维护性

| 指标 | 当前 | 合并后 |
|------|:---:|:-----:|
| 函数文件数 | 17 | **13** |

## 六、建议实施顺序

1. **第一阶段**：新建 3 个统一函数（`main-tick`, `apply/set_duration`, `desctuor/remove_effects`）
2. **第二阶段**：修改 trigger 函数（`arrow_hit/*`, `area_select/*`），移除转移逻辑，改引用新名称
3. **第三阶段**：修改 `tick.mcfunction`、`player_death/main.mcfunction`、`load.mcfunction`

5. **第五阶段**：在 `load.mcfunction` 中添加存量迁移逻辑

## 七、结论

**合并是可行的，且收益显著。**

两种定身类型的核心差异仅在于**初始时长计算**和**叠加时长计算**——这两个差异由各自的 trigger 函数内部处理，不依赖于标签或计分板的命名。其余所有逻辑（tick 递减、效果应用、效果清除、展示实体、最大次数限制）完全同构，可以安全合并。

最主要的风险是存量世界兼容性，但可以通过简单的 load 迁移函数解决。
